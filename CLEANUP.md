- [ ] **Cross-sell toast — two confirmed misbehaviors (Phase A.4 testing).**
  - **Bug A — fires when it should not.** The toast (`DashboardClient`) and banner (`upgrade/page.tsx`) show an "upgrade to Everything" prompt to users who *already hold both products' entitlements* — e.g. an Everything subscriber, with both `intelligence:pro` and `meetings:pro`. Confirmed on the Meetings dashboard for an Everything subscriber; the Intelligence side has the mirror-image bug. Fix: also check for the *absence* of the other product's entitlement — suppress when the user owns both products. ~5 lines per repo (one extra `getEntitlement(uid, OTHER_PRODUCT)` check; no new helper or `subscription_id` inspection — owning both is reason enough to suppress, bundle or not).
  - **Bug B — does not fire when it should.** A user with only the *other* product's entitlement (subscribed to Meetings Pro, then visited the Intelligence dashboard as a free Intelligence user) saw no cross-sell toast, though they hold `meetings:pro` and should be prompted to upgrade to Everything.
  - **Bug B suspect causes** — Intelligence repo grepped (`CrossSellToast` referenced in `components/CrossSellToast.tsx`, `dashboard/DashboardClient.tsx`, `dashboard/page.tsx`):
    - *Not wired* — **ruled out for Intelligence.** `CrossSellToast` is imported and rendered in `DashboardClient.tsx`, gated by the `hasOtherProduct` prop, which `dashboard/page.tsx` sets from `!!getEntitlement(uid, "meetings")`.
    - *Backwards / incomplete condition* — **ruled out for Intelligence.** The trigger checks the OTHER product (`getEntitlement(uid, "meetings")`), which is correct. The **Meetings** repo's `CrossSellToast` wiring/condition is a separate repo/session and still needs the same verification — cannot be checked from here.
    - *localStorage key collision* — **ruled out.** Intelligence's key is `cross_sell_toast_seen_meetings_v1` (per-product suffix present); Meetings' is `cross_sell_toast_seen_intelligence_v1`. localStorage is also per-origin (`app.haterz.ai` vs `meetings.haterz.ai`), so the two apps cannot collide regardless.
    - *localStorage key already burned* — **most likely cause.** `CrossSellToast` writes its key on *first render* (one-shot per browser, by step-9 design), not on dismiss/CTA. Any earlier Phase A test that rendered the Intelligence toast — including an Everything subscriber triggering **Bug A** on the Intelligence dashboard — already set `cross_sell_toast_seen_meetings_v1`, so it cannot show again in that browser. Confirm by clearing localStorage / using a fresh or incognito browser. Interaction worth noting: **Bug A firing burns the key, which then produces Bug B.**
    - *Meetings subscription not active* — **also possible.** `getEntitlement` filters to `status in ('active','trialing')`. If the test's Meetings Pro subscription is stuck at `incomplete` (the webhook race fixed in `04d1934` — only fixed when testing a branch that includes it), `getEntitlement(uid, "meetings")` correctly returns null → `hasOtherProduct` false → no toast. In that case the defect is the subscription status, not the toast. Verify the Meetings sub row's `status` in the DB.

Needed if out-of-band subscription creation is ever introduced (would require re-adding a `customer.subscription.created` handler), or to make status writes fully deterministic regardless of webhook delivery/commit order.

Wire to Sentry / log alerting when production observability lands.
